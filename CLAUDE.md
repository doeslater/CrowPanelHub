# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

"CrowPanelHub" is an exploration project for the **CrowPanel ESP32 4.2" e-paper HMI display** (see Sources below). The explicit goal is to understand the hardware and **help other developers dive into the ESP32 world** — this is a teaching/learning project, not a polished product. It's a public repo pushed to `github.com/doeslater/CrowPanelHub` with **no `LICENSE` file** — a deliberate choice to keep it unencumbered rather than adding MIT boilerplate, though note this legally defaults to all-rights-reserved rather than true public domain (an explicit public-domain dedication like The Unlicense/CC0 would be needed for that, if it's ever wanted). The Android app (package `com.example.crowpanelhub`) is the companion app that sends images to the panel over USB serial — milestone 1's send path is built and hardware-verified (see Current milestone below).

**Current state — read this before trusting anything else in this file about "current" architecture:**
- `app/` is no longer a bare template. It implements milestone 1's USB-serial connect/send flow end to end, plus a Diagnostics screen (see Architecture below for the actual files) — two Compose screens wired together via `androidx-navigation-compose`, Hilt for DI. What's *not* built yet: the `Transport` interface abstraction and Wi-Fi/BLE — those remain as described in the Transport & wire protocol section below.
- `display_text/` is a standalone Arduino `.ino` sketch (not part of the Android build, not using PlatformIO) adapted from a third-party Elecrow example. It renders a fixed set of hardcoded text rows to the panel on boot. It is **not** the milestone-1 image-transfer firmware described below — no serial receive loop, no USB protocol, nothing wired to the phone.

Do not assume any BLE/Wi-Fi code or repository/data-layer code exists in this repo until it's actually been added — check the current files before referencing them, since parts of this document still describe *planned* design for what isn't built yet (clearly marked as such below).

A `MEMORY.md` file at the repo root (gitignored, local-only — not on GitHub) holds the narrative/rationale behind earlier planning decisions, if present on this machine. Note it was written before the app and firmware were reset to a bare-scaffold state (since built back up into what's described above) — treat its description of "current" code as intent/history, not present fact. Don't assume it exists on a fresh clone.

## Hardware target

- **Board**: CrowPanel ESP32 4.2" E-Paper HMI Display, SKU DIE07300S (Elecrow), ESP32-S3-WROOM-1-N8R8.
- **Panel**: 400×300, black/white only, driven by SPI, SSD1683 controller.
- **Firmware ecosystem**: Arduino-based; Elecrow ships demo/example sketches, and the GxEPD2 library is the common driver for this panel (see Sources). The existing `display_text/` sketch uses `GxEPD2_BW<GxEPD2_420_GYE042A87, ...>` from that library.
- **Firmware lives in this repo**: ESP32 firmware is authored as a subproject alongside `app/`, as a sibling directory at the repo root (e.g. `display_text/`), not nested inside the Android module.

## Current milestone

**Milestone 1**: send a bitmap from the phone to the ESP32 over USB serial and see it render on the panel, using a full (not partial) e-paper refresh. This is fire-and-forget: the ESP32 does not send anything back, and Android considers the send successful once the bytes are written to the serial port. Visual confirmation is by looking at the panel during bring-up. **Done, on both sides, and hardware-verified**: the firmware (`receive_image/`) receives/verifies/renders the wire-protocol frame; the Android app connects over USB serial and can send either a programmatically-generated checkerboard (`Checkerboard.kt` — a small loop building packed 1-bit bytes, no bundled binary asset, so the code stays readable/teachable) or rendered demo text (`TextBitmap.kt`, drawn via Android's `Canvas`/`Paint`), via two separate buttons — "Send Text" and "Send Checkerboard" — rather than the single "Send (test pattern)" button originally planned (see Architecture below for why both were kept). Both patterns have been confirmed rendering correctly on a real CrowPanel board.

**Milestone 2** (after milestone 1 works): swap the checkerboard for a QR code (via a library such as ZXing) that links to information about the project/how it was built — the panel becomes something a person could scan and learn from. The QR's target URL is not yet decided.

USB serial was chosen to start (over Wi-Fi/BLE) because it needs no pairing/provisioning UI, giving the tightest feedback loop while the ESP32-side rendering path is still being proven out.

## Transport & wire protocol

- **Transport abstraction**: planned for when a second transport (Wi-Fi/BLE) actually exists, but deliberately **not built yet**. `MainViewModel` currently depends directly on the concrete `UsbSerialTransport` class (Hilt-injected) rather than an interface — an explicit choice favoring simplicity/directness (per this project's stated priority: code simple enough for someone with basic coding skills to follow) over speculative future-proofing for a transport that doesn't exist yet. When Wi-Fi/BLE work starts, introduce a thin `interface Transport { suspend fun sendImage(bytes: ByteArray): Result<Unit>; val connectionState: StateFlow<ConnectionState> }` then, motivated by a real second implementation rather than built in advance.
- **Build order**: USB serial first (done), then Wi-Fi, then BLE (both not started).
- **Single active transport**: only one transport is ever connected at a time — trivially true right now since USB is the only one that exists. Once Wi-Fi/BLE exist, the user will explicitly pick which one — no simultaneous multi-transport support planned.
- **Android USB serial library**: implemented — `com.github.mik3y:usb-serial-for-android` (JitPack, version `3.10.0`) rather than raw `UsbManager`/CDC-ACM handling, used in `UsbSerialTransport.kt`. The CrowPanel board was confirmed via `lsusb` during firmware bring-up to use a CH340 bridge chip; this library's default prober (`UsbSerialProber.getDefaultProber()`) also handles CP210x/CH9102/FTDI/PL2303 and native CDC-ACM, so it isn't hardcoded to CH340 specifically.
- **Connect UX**: implemented as planned — a manual "Connect" button (`MainAction.OnConnectClick`), no auto-launch via a `USB_DEVICE_ATTACHED` intent-filter, no hardcoded VID/PID. The permission flow bridges Android's async USB-permission dialog into a suspend function (`UsbSerialTransport`'s private `requestPermission`, via `BroadcastReceiver` + `PendingIntent`), with status visible in the UI (`MainState.statusText`) rather than manifest magic.
- **Screen layout**: `MainScreen` hosts a "Diagnostics" button (navigates to the Diagnostics screen below), the transport picker (a `SingleChoiceSegmentedButtonRow`, currently showing only "USB" since no other transport exists yet), the Connect button, and — deviating from the single "Send (test pattern)" button originally planned — **two** Send buttons ("Send Text" / "Send Checkerboard"), since both patterns turned out worth keeping for experimentation rather than picking just one. A status row shows a `CircularProgressIndicator` plus status text while a send is in flight, driven by a simulated progress estimate (see `MainViewModel` in Architecture below) since the fire-and-forget protocol gives no real completion signal.
- **Diagnostics screen**: implemented — a permanent, user-facing, **read-only** screen (`DiagnosticsScreen.kt`/`DiagnosticsViewModel.kt`) showing live connection status, last error, and a scrolling event log (newest first) sourced from `UsbSerialTransport.logEntries`. Reachable via a "Diagnostics" button on `MainScreen`, not from a Settings screen as originally sketched — there's no Settings screen in this app, and connections are still only ever initiated from `MainScreen`.
- **Wire protocol** (phone → ESP32 only, one-directional): implemented and hardware-verified — a minimal length-prefixed binary frame `[1-byte magic][4-byte little-endian payload length][4-byte little-endian epoch seconds][payload: packed 1-bit bitmap, 400×300 = 15,000 bytes][1-byte checksum]`. Magic byte is `0xA5`; checksum is an 8-bit sum (mod 256) of the payload bytes only. The timestamp exists so the panel can display a "last updated" date/time label — the ESP32 has no RTC and no Wi-Fi/NTP in milestone 1, so it cannot know the time on its own, and firmware (`receive_image.ino`) does **zero timezone math**: it just formats whatever epoch value it receives, as-is, assuming it's already local wall-clock time. The Android sender (`WireFrame.kt`) exploits this: instead of sending true UTC epoch seconds plus a separate timezone-offset field, it pre-adds the phone's current local UTC offset (`TimeZone.getDefault().getOffset(...)`) directly into the epoch value before sending, so the firmware's naive formatting produces the phone's correct local time with **zero protocol change and zero firmware timezone-awareness**. (An earlier iteration added a real 2-byte timezone-offset field to the wire format plus matching firmware/Python-script changes — that was reverted in favor of this simpler, Android-only approach; a `TZ_OFFSET_SIZE` field does not exist if you see it mentioned in old context.)
- **Baud rate**: 115200 — the universal ESP32/Arduino default (also what the existing `display_text/` sketch uses for its `Serial` logging), and more than fast enough (~1.3s per frame) next to the multi-second e-paper refresh itself.
- **Firmware refresh mode**: full refresh only for milestone 1 (`epd.display()`), not partial refresh — simplest, matches every GxEPD2 example (including `display_text/`, which calls `epd.init(..., partial = false)`), and ghosting/speed concerns from partial refresh aren't relevant when sending one deliberate frame at a time.
- **Firmware toolchain**: PlatformIO (CLI-first: `pio run`, `pio run -t upload`, `pio device monitor`), not the Arduino IDE — chosen because it's scriptable from the terminal and gives version-pinned library deps via `platformio.ini`, mirroring `libs.versions.toml` on the Android side. **Not installed on this machine yet**, and the current `display_text/`/`receive_image/` sketches are plain `.ino`/`config.h` pairs with no `platformio.ini` — migrating either to PlatformIO is still pending.
- Writing up findings/notes from this exploration for other developers is an out-of-repo activity (docs/blog-style), not an in-app feature — don't build a "share log" or similar mechanism unless explicitly asked again.

## Commands

All commands run from the repo root via the Gradle wrapper.

- Build debug APK: `./gradlew assembleDebug`
- Run local unit tests (JVM, in `app/src/test`): `./gradlew testDebugUnitTest`
- Run a single unit test: `./gradlew testDebugUnitTest --tests "com.example.crowpanelhub.ExampleUnitTest"`
- Run instrumented/UI tests (in `app/src/androidTest`, requires a connected device/emulator): `./gradlew connectedDebugAndroidTest`
- Run Android Lint: `./gradlew lint`
- Install debug build on a connected device: `./gradlew installDebug`

There is no ktlint/detekt/spotless config and no CI workflow in this repo — lint/test enforcement is whatever you invoke manually via Gradle. Both `app/src/test/.../ExampleUnitTest.kt` and `app/src/androidTest/.../ExampleInstrumentedTest.kt` are still the unmodified Android Studio template tests.

There is no firmware build tooling yet (no PlatformIO install, no `platformio.ini`); `display_text/display_text.ino` is currently only buildable/flashable via the Arduino IDE (with the GxEPD2 library and its font headers installed). See `docs/arduino-ide-setup.md` for a known-working board configuration (from a successful upload, transcribed from a screenshot).

## Architecture (current code)

- **`app/`**: implements milestone 1's Android side (USB-serial connect + send) plus the Diagnostics screen, hardware-verified. Single Gradle module, everything flat in the `com.example.crowpanelhub` root package (no `data`/`viewmodel`/`navigation` subpackages) — a deliberate simplicity choice at this project size, same reasoning as skipping the `Transport` interface (see Transport & wire protocol above). Hilt for DI; `androidx-navigation-compose` provides a two-route `NavHost` (`"main"`/`"diagnostics"`) hosted directly in `MainActivity.kt` rather than a separate nav-graph file, since there are only two screens so far.
  - `CrowPanelHubApp.kt`: `@HiltAndroidApp` `Application` class, registered in the manifest as `android:name=".CrowPanelHubApp"`.
  - `MainActivity.kt`: `@AndroidEntryPoint` `ComponentActivity` hosting a `NavHost` (routes `"main"`/`"diagnostics"`, via `rememberNavController()`) inside a `Scaffold`, rendering `MainRoot()`/`DiagnosticsRoot()`. The template's `Greeting`/`GreetingPreview` composables are gone, fully replaced.
  - `MainViewModel.kt`: `@HiltViewModel`, MVI-style (`MainState`/`MainAction`/`MainViewModel` — no `Event` type, since navigating to Diagnostics is handled by a plain callback (`onNavigateToDiagnostics`) passed into `MainScreen`, not a one-time emitted event). Observes `UsbSerialTransport.connectionState` and maps it to `MainState.statusText`/`isConnected`/`isBusy`. `onAction` handles `OnTransportSelected`, `OnConnectClick` (calls `UsbSerialTransport.connect()`), `OnSendTextClick` (sends `TextBitmap.generate()`), and `OnSendCheckerboardClick` (sends `Checkerboard.generate()`) — the two send actions share a private `send(payload)` helper, which also drives `MainState.progressFraction`: since the wire protocol is fire-and-forget with no completion signal from the ESP32, it ticks a simulated progress bar across an estimated write-time-plus-panel-refresh window (~1.3s write + ~4s cooldown) rather than reporting real progress.
  - `MainScreen.kt`: `MainRoot`/`MainScreen` composables (Root + Screen in the same file, per convention) — a "Diagnostics" button (calls `onNavigateToDiagnostics`), a `SingleChoiceSegmentedButtonRow` transport picker (currently just "USB"), a Connect button, "Send Text"/"Send Checkerboard" buttons (enabled only once connected and not mid-operation), and a status row (`CircularProgressIndicator` while `isBusy`, plus status text). Includes a `@Preview`.
  - `UsbSerialTransport.kt`: `@Singleton`, Hilt-injected with `@ApplicationContext`. Talks to the board via `usb-serial-for-android`'s `UsbSerialProber`. `connect()` finds the device, requests USB permission if not already granted (bridging Android's async permission dialog into a suspend function via `BroadcastReceiver` + `PendingIntent` — `FLAG_MUTABLE` and `ContextCompat.registerReceiver(..., RECEIVER_NOT_EXPORTED)` both matter here and are easy to get subtly wrong, see inline comments), opens the port at 115200/8N1, and exposes `connectionState: StateFlow<UsbConnectionState>` (`Disconnected`/`RequestingPermission`/`PermissionDenied`/`Connecting`/`Connected`/`ConnectFailed`). `sendImage(payload)` builds the wire frame and writes it. No `disconnect()`/reconnect handling exists — a documented gap, not an oversight, since there's no Disconnect button in this design. Also owns `logEntries: StateFlow<List<LogEntry>>` (capped at 100, each a timestamp + message), appending an entry on every state transition and send attempt/result — this feeds the Diagnostics screen's event log. `UsbConnectionState.toStatusText()` used to be private to `MainViewModel`; it's now a top-level function here so both `MainViewModel` and `DiagnosticsViewModel` can share it.
  - `DiagnosticsViewModel.kt`: `@HiltViewModel`, read-only — no `Action`/`Event` type, since this screen only displays state (`DiagnosticsState`: `statusText`, `lastErrorText`, `logEntries`) and has nothing to trigger. Observes `UsbSerialTransport.connectionState` (mapped via the shared `toStatusText()`, with `lastErrorText` set for `ConnectFailed`/`PermissionDenied`) and `UsbSerialTransport.logEntries` directly.
  - `DiagnosticsScreen.kt`: `DiagnosticsRoot`/`DiagnosticsScreen` composables — status text, last error (if any), and a `LazyColumn` event log (newest first, each row timestamped via `SimpleDateFormat("HH:mm:ss")`). Includes a `@Preview`.
  - `WireFrame.kt`: pure Kotlin, builds the wire-protocol frame (see Transport & wire protocol above) — owns the `DISPLAY_WIDTH`/`DISPLAY_HEIGHT`/`PAYLOAD_SIZE` constants (shared by both bitmap generators below) and the local-time-epoch trick.
  - `TextBitmap.kt`: draws demo text onto a 400×300 `Canvas` (via `Paint`) and packs the result into the same 1bpp format `WireFrame` expects — the phone-side equivalent of `display_text.ino`'s hardcoded rows, but sent as a plain bitmap over the wire protocol rather than rendered firmware-side. Wired to the "Send Text" button.
  - `Checkerboard.kt`: the original milestone-1 test pattern (matches `receive_image/send_test_frame.py`'s algorithm), wired to the "Send Checkerboard" button. Kept alongside `TextBitmap` deliberately as a second known-good pattern for future experimentation, not dead code.
  - `ui/theme/{Color,Theme,Type}.kt`: default Material 3 theme (`CrowPanelHubTheme`), unmodified template colors/typography, dynamic color enabled on API 31+.
- **`display_text/`** (repo root, sibling of `app/`): a standalone Arduino sketch, not wired into the Gradle build. Placed here deliberately as a **reference example** (GxEPD2 init/pin-mapping/power-cycle patterns) — do not modify or delete it, and do not treat it as a spike to evolve into the milestone-1 firmware; write new sketch files for that instead, consulting this one for patterns.
  - `display_text.ino`: defines a `DisplaySettings`/`Row` model (up to 8 text rows, per-row font size, optional border/color-inversion), a `GxEPD2_BW<GxEPD2_420_GYE042A87, ...>` display instance on pins `PWR=7 BUSY=48 RES=47 DC=46 CS=45`, and an `updateDisplay()` that powers the panel, draws the configured rows, calls `epd.display()`, then hibernates and powers down. `setup()` hardcodes 5 rows of demo text ("Hello World" / "Second line" / ...) and calls `updateDisplay()` once; `loop()` just delays — no serial input handling, no dynamic content.
  - `config.h`: compile-time constants (display size 400×300, margin, baud rate, font sizes, row heights).
- **`receive_image/`** (repo root, sibling of `app/` and `display_text/`): the milestone-1 firmware sketch — a serial receive loop, not present in `display_text/`. Compiles for `esp32:esp32:esp32s3` and **has been flashed to and verified against real hardware** (a CrowPanel board on `/dev/ttyUSB0`, CH340 bridge chip), both via hand-written Python scripts (below, sending true UTC) and the real Android app (sending phone-local time disguised as epoch seconds — see Transport & wire protocol above) — magic byte, length, timestamp, checksum, and rendering all confirmed working end-to-end from both senders.
  - `receive_image.ino`: scans incoming `Serial` bytes for the magic byte, then reads the length-prefixed frame described in the Wire protocol bullet above, verifies the checksum, and renders via `epd.drawBitmap(...)` for the bitmap plus a `strftime`-formatted "last updated" date/time label drawn over the bottom of the frame (formatted as-is, no timezone math — see above), then `epd.display()` using the same power-cycle (`epdPower`/`epd.init`/`epd.hibernate`) pattern as `display_text.ino`. Any malformed/mismatched frame is dropped with a `Serial.println` and no response sent — fire-and-forget, matching the milestone-1 spec.
  - `config.h`: pin definitions (same as `display_text/`), display size, and the wire-protocol constants (magic byte, payload size, timestamp field size, baud rate, serial timeout).
  - `send_test_frame.py`: the Python script standing in for the Android sender during firmware bring-up (before the Android app existed) — stdlib-only (no `pyserial`), resets the board via raw `termios`/`fcntl` DTR/RTS control, builds a checkerboard-pattern frame with the current **true UTC** time via `build_frame(payload)` (unlike the Android app, this script never adds a local offset), sends it, and prints the board's `Serial` responses. Exposes `send_payload(payload)` so other scripts can reuse the reset/send/report logic with a different bitmap.
  - `send_text.py`: same wire protocol and board, different payload — renders arbitrary text (CLI args, or built-in default lines) into a 400×300 bitmap with Pillow and sends it via `send_payload(...)` from `send_test_frame.py`. No firmware change needed, since `receive_image.ino` just draws whatever bitmap it's given. Needs Pillow installed (`pip install pillow`), unlike `send_test_frame.py`.

## Build configuration notes

- Kotlin 2.2.10, AGP 9.2.1, Gradle 9.4.1 (see `gradle/wrapper/gradle-wrapper.properties`).
- compileSdk/targetSdk 37, minSdk 24. `sourceCompatibility`/`targetCompatibility` are JavaVersion 11 (not 17).
- Compose compiler is applied via the `org.jetbrains.kotlin.plugin.compose` plugin (Kotlin 2.x style), not the old `composeOptions.kotlinCompilerExtensionVersion`.
- **Hilt** (`com.google.dagger.hilt.android`, version `2.59.2`) for DI, with **KSP** (`com.google.devtools.ksp`, version `2.3.9`) for annotation processing — not kapt; use the `hilt-compiler` artifact with `ksp(...)`, not the kapt-era `hilt-android-compiler`. KSP's version here is **not** tied 1:1 to the Kotlin version — KSP switched to a standalone versioning scheme at `2.3.0`. `2.3.9` specifically was required because older KSP builds paired to Kotlin 2.2.10 (e.g. `2.2.10-2.0.2`) don't correctly support AGP 9's "built-in Kotlin" compilation (`google/ksp#2729`); if bumping Kotlin or KSP later, re-verify built-in-Kotlin compatibility rather than assuming an old pairing still works — a mismatch here fails in a confusing way (`compileDebugKotlin`/`kspDebugKotlin` tasks silently don't run, so the build reports success while actually producing an APK missing all Kotlin-compiled classes).
- **usb-serial-for-android** (`com.github.mik3y:usb-serial-for-android`, version `3.10.0`) is fetched from **JitPack** — `maven(url = "https://jitpack.io")` is added to `settings.gradle.kts`'s `dependencyResolutionManagement.repositories`, not inside `app/build.gradle.kts` (a per-module `repositories {}` block there is disallowed by this project's `FAIL_ON_PROJECT_REPOS` mode).
- **androidx-navigation-compose** (version `2.9.8`) provides the `NavHost` wiring `MainScreen`/`DiagnosticsScreen` together — added alongside the pre-existing `androidx-hilt-navigation-compose` (which was already present for `hiltViewModel()` regardless of navigation).
- No `kotlinx.serialization` plugin is applied anywhere in the project yet.
- Dependency versions are centralized in `gradle/libs.versions.toml` (version catalog) — add new dependencies there rather than hardcoding coordinates in `app/build.gradle.kts`.
