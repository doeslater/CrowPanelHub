# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Working style

Before taking any action (running a command, flashing firmware, sending a serial frame, editing a file), state the plan and wait for an explicit go-ahead ã don't chain an action onto an answer on your own initiative, even a small or reversible one. Answering questions or reading/explaining code doesn't need this.

For multi-step or multi-file work, state the full plan up front (which files, what changes, how it'll be verified) and get **one** explicit go-ahead for the whole batch, rather than checking in after each individual file ã then execute without stopping again mid-way. Answering a clarifying/design question (e.g. picking between options) is **not** itself a go-ahead to start implementing ã a separate, explicit go-ahead is still required before execution begins. While executing an approved plan, narrate the reasoning behind choices as they're made (why this approach, why this file, what a step revealed) rather than just reporting a log of actions taken afterward.

### Teaching mode (`workspace/exercises/`)

The user is a self-declared ESP32/hardware beginner whose actual goal for this project is to learn to write ESP32 firmware themselves, not to have it written for them. An earlier attempt at teaching-by-explaining (`workspace/sketches/receive_image/`, since removed) failed ã it was "not clear at all" ã and a follow-up pattern of dumping full code walkthroughs on request ("crunching code") made things worse, not better. The rules below are the fix, and they **replace** the general "I do the work, you approve" model above, scoped specifically to the `workspace/exercises/` directory. Everything else in the repo (`workspace/sketches/`, `app/`, docs, ordinary maintenance work) still follows the Working style rules above, unchanged ã this is not a general policy of doing less; it's specific to the learning track.

- **The user writes and runs everything.** Don't write exercise code or run `arduino-cli`/`install.sh` on the user's behalf inside `workspace/exercises/` ã typing the code and running the compile/upload command themselves (including reading their own compiler errors) is the point. The job here is reviewing, questioning, and hinting ã not authoring or executing.
- **Socratic by default**, for both plain explanations and hands-on checkpoints. When asked "what does this do," don't answer in full ã ask what they think a specific piece does first, one piece at a time, and build from their answer. This applies to descriptive/comprehension questions too, not just exercises.
- **Theory before every checkpoint.** Each new concept gets a short explanation (a few sentences to a paragraph, not a lecture) of *why* it works, before any hands-on attempt ã enough to attempt the checkpoint with understanding, not so much that the checkpoint becomes redundant.
- **One hint after a real attempt, not before.** If a checkpoint fails, let the user actually try first; only then give exactly one hint sized to unstick them, never the answer outright.
- **Everything stays tied to the real CrowPanel board** ã no generic off-board filler exercises (e.g. a plain-Arduino "blink an LED" tutorial, when this board has no onboard LED). Find this board's native equivalent of any generic concept instead ã e.g. toggling `PWR` (the panel's power-switch pin) as the CrowPanel-native version of "blink an LED."
- **Curriculum**: exercises are drawn from `docs/ideas.md`'s 30-idea pool, sequenced by merging its Easy/Medium/Hard tiers with its "natural sequence" dependency ordering into one simple-to-complex learning curve ã see `docs/firmware-learning-path.md` for the actual sequence this produces.
- **Progress tracking**: `workspace/exercises/PROGRESS.md` records the current concept/exercise and its status (theory given / attempted / hint given / passed). Read it at the start of any `workspace/exercises/` work, rather than assuming where the user is ã especially important after a `/clear`, when this conversation's context is gone but their progress isn't.

## Project overview

"CrowPanelHub" is an exploration project for the **CrowPanel ESP32 4.2" e-paper HMI display** (see Sources below). The explicit goal is to understand the hardware and **help other developers dive into the ESP32 world** ã this is a teaching/learning project, not a polished product. It's a public repo pushed to `github.com/doeslater/CrowPanelHub` with **no `LICENSE` file** ã a deliberate choice to keep it unencumbered rather than adding MIT boilerplate, though note this legally defaults to all-rights-reserved rather than true public domain (an explicit public-domain dedication like The Unlicense/CC0 would be needed for that, if it's ever wanted). The Android app (package `com.example.crowpanelhub`) is the companion app that sends images to the panel over USB serial ã milestone 1's send path is built and hardware-verified (see Current milestone below).

**Current state ã read this before trusting anything else in this file about "current" architecture:**
- `app/` is no longer a bare template. It implements milestone 1's USB-serial connect/send flow end to end, plus a Diagnostics screen (see Architecture below for the actual files) ã two Compose screens wired together via `androidx-navigation-compose`, Hilt for DI. What's *not* built yet: the `Transport` interface abstraction and Wi-Fi/BLE ã those remain as described in the Transport & wire protocol section below.
- `workspace/sketches/display_text/` is a standalone Arduino `.ino` sketch (not part of the Android build, not using PlatformIO) adapted from a third-party Elecrow example. It renders a fixed set of hardcoded text rows to the panel on boot. It is **not** the milestone-1 image-transfer firmware described below ã no serial receive loop, no USB protocol, nothing wired to the phone.

Do not assume any BLE/Wi-Fi code or repository/data-layer code exists in this repo until it's actually been added ã check the current files before referencing them, since parts of this document still describe *planned* design for what isn't built yet (clearly marked as such below).

A `MEMORY.md` file at the repo root (gitignored, local-only ã not on GitHub) holds the narrative/rationale behind earlier planning decisions, if present on this machine. Note it was written before the app and firmware were reset to a bare-scaffold state (since built back up into what's described above) ã treat its description of "current" code as intent/history, not present fact. Don't assume it exists on a fresh clone.

## Hardware target

See `HARDWARE.md` (repo root, gitignored/local-only) for canonical device names, MAC addresses, and port/bridge-chip identity ã this section covers firmware-role and architecture context only, not device identity.

### CrowPanel
- **Panel**: 400û300, black/white only, driven by SPI, SSD1683 controller.
- **Firmware ecosystem**: Arduino-based; Elecrow ships demo/example sketches, and the GxEPD2 library is the common driver for this panel (see Sources). The existing `workspace/sketches/display_text/` sketch uses `GxEPD2_BW<GxEPD2_420_GYE042A87, ...>` from that library.
- **Firmware lives in this repo**: ESP32 firmware is authored as a subproject alongside `app/`, under `workspace/sketches/`. `workspace/` is a repo-root sibling of `app/` holding firmware sketches and the `workspace/exercises/` teaching curriculum together (kept separate from `app/` for tidiness, since this is fundamentally an Android project with firmware as a secondary concern), with each sketch as its own subfolder under `workspace/sketches/` (e.g. `workspace/sketches/display_text/`) ã not nested inside the Android module.

### basic board
Used for firmware sketches in [workspace/basic_board](workspace/basic_board), especially [workspace/basic_board/pwr_pin](workspace/basic_board/pwr_pin) and [workspace/basic_board/rgb_led](workspace/basic_board/rgb_led). This is a teaching companion to the CrowPanel board, not a replacement for it.


## Current milestone

**Milestone 1**: send a bitmap from the phone to the ESP32 over USB serial and see it render on the panel, using a full (not partial) e-paper refresh. This is fire-and-forget: the ESP32 does not send anything back, and Android considers the send successful once the bytes are written to the serial port. Visual confirmation is by looking at the panel during bring-up. **Done, on both sides, and hardware-verified**: the receiving firmware (originally `workspace/sketches/receive_image/`, since removed ã it wasn't a clear teaching vehicle, see Architecture below) received/verified/rendered the wire-protocol frame; `workspace/sketches/test_card/test_card.ino` implements the same wire protocol today and remains hardware-verified doing so. The Android app connects over USB serial and can send either a programmatically-generated checkerboard (`Checkerboard.kt` ã a small loop building packed 1-bit bytes, no bundled binary asset, so the code stays readable/teachable) or rendered demo text (`TextBitmap.kt`, drawn via Android's `Canvas`/`Paint`), via two separate buttons ã "Send Text" and "Send Checkerboard" ã rather than the single "Send (test pattern)" button originally planned (see Architecture below for why both were kept). Both patterns have been confirmed rendering correctly on a real CrowPanel board.

USB serial was chosen to start (over Wi-Fi/BLE) because it needs no pairing/provisioning UI, giving the tightest feedback loop while the ESP32-side rendering path is still being proven out.

## Transport & wire protocol

- **Transport abstraction**: planned for when a second transport (Wi-Fi/BLE) actually exists, but deliberately **not built yet**. `MainViewModel` currently depends directly on the concrete `UsbSerialTransport` class (Hilt-injected) rather than an interface ã an explicit choice favoring simplicity/directness (per this project's stated priority: code simple enough for someone with basic coding skills to follow) over speculative future-proofing for a transport that doesn't exist yet. When Wi-Fi/BLE work starts, introduce a thin `interface Transport { suspend fun sendImage(bytes: ByteArray): Result<Unit>; val connectionState: StateFlow<ConnectionState> }` then, motivated by a real second implementation rather than built in advance.
- **Build order**: USB serial first (done), then Wi-Fi, then BLE (both not started).
- **Single active transport**: only one transport is ever connected at a time ã trivially true right now since USB is the only one that exists. Once Wi-Fi/BLE exist, the user will explicitly pick which one ã no simultaneous multi-transport support planned.
- **Android USB serial library**: implemented ã `com.github.mik3y:usb-serial-for-android` (JitPack, version `3.10.0`) rather than raw `UsbManager`/CDC-ACM handling, used in `UsbSerialTransport.kt`. The CrowPanel board was confirmed via `lsusb` during firmware bring-up to use a CH340 bridge chip; this library's default prober (`UsbSerialProber.getDefaultProber()`) also handles CP210x/CH9102/FTDI/PL2303 and native CDC-ACM, so it isn't hardcoded to CH340 specifically.
- **Connect UX**: implemented as planned ã a manual "Connect" button (`MainAction.OnConnectClick`), no auto-launch via a `USB_DEVICE_ATTACHED` intent-filter, no hardcoded VID/PID. The permission flow bridges Android's async USB-permission dialog into a suspend function (`UsbSerialTransport`'s private `requestPermission`, via `BroadcastReceiver` + `PendingIntent`), with status visible in the UI (`MainState.statusText`) rather than manifest magic.
- **Screen layout**: `MainScreen` hosts a "Diagnostics" button (navigates to the Diagnostics screen below), the transport picker (a `SingleChoiceSegmentedButtonRow`, currently showing only "USB" since no other transport exists yet), the Connect button, and ã deviating from the single "Send (test pattern)" button originally planned ã **two** Send buttons ("Send Text" / "Send Checkerboard"), since both patterns turned out worth keeping for experimentation rather than picking just one. A status row shows a `CircularProgressIndicator` plus status text while a send is in flight, driven by a simulated progress estimate (see `MainViewModel` in Architecture below) since the fire-and-forget protocol gives no real completion signal.
- **Diagnostics screen**: implemented ã a permanent, user-facing, **read-only** screen (`DiagnosticsScreen.kt`/`DiagnosticsViewModel.kt`) showing live connection status, last error, and a scrolling event log (newest first) sourced from `UsbSerialTransport.logEntries`. Reachable via a "Diagnostics" button on `MainScreen`, not from a Settings screen as originally sketched ã there's no Settings screen in this app, and connections are still only ever initiated from `MainScreen`.
- **Wire protocol** (phone ã ESP32 only, one-directional): implemented and hardware-verified ã a minimal length-prefixed binary frame `[1-byte magic][4-byte little-endian payload length][4-byte little-endian epoch seconds][payload: packed 1-bit bitmap, 400û300 = 15,000 bytes][1-byte checksum]`. Magic byte is `0xA5`; checksum is an 8-bit sum (mod 256) of the payload bytes only. The timestamp exists so the panel can display a "last updated" date/time label ã the ESP32 has no RTC and no Wi-Fi/NTP in milestone 1, so it cannot know the time on its own, and firmware (`test_card.ino`) does **zero timezone math**: it just formats whatever epoch value it receives, as-is, assuming it's already local wall-clock time. The Android sender (`WireFrame.kt`) exploits this: instead of sending true UTC epoch seconds plus a separate timezone-offset field, it pre-adds the phone's current local UTC offset (`TimeZone.getDefault().getOffset(...)`) directly into the epoch value before sending, so the firmware's naive formatting produces the phone's correct local time with **zero protocol change and zero firmware timezone-awareness**. (An earlier iteration added a real 2-byte timezone-offset field to the wire format plus matching firmware/Python-script changes ã that was reverted in favor of this simpler, Android-only approach; a `TZ_OFFSET_SIZE` field does not exist if you see it mentioned in old context.)
- **Baud rate**: 115200 ã the universal ESP32/Arduino default (also what the existing `workspace/sketches/display_text/` sketch uses for its `Serial` logging), and more than fast enough (~1.3s per frame) next to the multi-second e-paper refresh itself.
- **Firmware refresh mode**: full refresh only for milestone 1 (`epd.display()`), not partial refresh ã simplest, matches every GxEPD2 example (including `workspace/sketches/display_text/`, which calls `epd.init(..., partial = false)`), and ghosting/speed concerns from partial refresh aren't relevant when sending one deliberate frame at a time.
- **Firmware toolchain**: `arduino-cli` is installed and is the actual tool used to compile/upload/monitor `workspace/sketches/display_text/` and `workspace/sketches/test_card/` (see `docs/dev-tools.md` for the exact commands, including the FQBN board-option string). PlatformIO remains the aspirational long-term choice (scriptable, version-pinned deps via `platformio.ini`, mirroring `libs.versions.toml`) but is **not installed on this machine**, and no `platformio.ini` exists yet ã migrating either sketch to PlatformIO is still pending. Don't assume there's no working CLI toolchain just because PlatformIO isn't set up; `arduino-cli` already covers compile/upload/monitor today.
- **`docs/fqbn.txt`**: the FQBN board-option string (`esp32:esp32:esp32s3:USBMode=hwcdc,CDCOnBoot=default,PSRAM=opi,PartitionScheme=default`) ã one line, no comments, kept for the manual `arduino-cli` walkthrough in `docs/dev-tools.md` (`$(cat docs/fqbn.txt)`) and as a reference copy. **Not** read by `install.sh` ã each one hardcodes its own copy of this same string instead (see below), so a sketch folder copied out on its own still works without this file. If the board's FQBN ever changes, update all three copies (this file plus the two `install.sh`s).
- **`install.sh`**: every firmware folder (`workspace/sketches/display_text/`, `workspace/sketches/test_card/`) has one ã compiles and uploads that folder's sketch (`./install.sh [port]`, defaulting to `/dev/ttyUSB0`), with the FQBN hardcoded directly in the script (not read from `docs/fqbn.txt`) so the folder works standalone if copied out of the repo. Identical content in both folders (it derives its own sketch directory from its own location, so nothing folder-specific needs to differ) ã same self-contained-copy convention as `config.h`.
- **`workspace/clear-cache.sh`**: clears `arduino-cli`'s compiled-sketch cache (`~/.cache/arduino/sketches/`, keyed by a hash of each sketch's source *path*), a sibling of `sketches/`, `basic_board/`, and `exercises/` at the `workspace/` root ã relevant to every firmware folder under `workspace/`, not just the original two sketches it was written alongside (hence living one level up from `sketches/`, not inside it). Clears the whole cache unconditionally (simplest ã costs one slower rebuild next compile) rather than trying to detect which entries are stale. Motivated by the `sketches/` move itself: each sketch's old repo-root path left an orphaned cache entry behind once its new `sketches/<name>/` path got its own.
- Writing up findings/notes from this exploration for other developers is an out-of-repo activity (docs/blog-style), not an in-app feature ã don't build a "share log" or similar mechanism unless explicitly asked again.

## Commands

All commands run from the repo root via the Gradle wrapper.

- Build debug APK: `./gradlew assembleDebug`
- Run local unit tests (JVM, in `app/src/test`): `./gradlew testDebugUnitTest`
- Run a single unit test: `./gradlew testDebugUnitTest --tests "com.example.crowpanelhub.ExampleUnitTest"`
- Run instrumented/UI tests (in `app/src/androidTest`, requires a connected device/emulator): `./gradlew connectedDebugAndroidTest`
- Run Android Lint: `./gradlew lint`
- Install debug build on a connected device: `./gradlew installDebug`

There is no ktlint/detekt/spotless config and no CI workflow in this repo ã lint/test enforcement is whatever you invoke manually via Gradle. Both `app/src/test/.../ExampleUnitTest.kt` and `app/src/androidTest/.../ExampleInstrumentedTest.kt` are still the unmodified Android Studio template tests.

Firmware is built/flashed via `arduino-cli`, not the Arduino IDE and not PlatformIO (no `platformio.ini` exists yet) ã see `docs/dev-tools.md` for the exact compile/upload/monitor commands and the FQBN board-option string, and `docs/arduino-ide-setup.md` for the underlying known-working board configuration (from a successful upload, transcribed from a screenshot) that those `arduino-cli` flags translate.

- Lint the two `workspace/basic_board/` sketches: `./workspace/lint.sh` ã compiles `basic_board/pwr_pin` and `basic_board/rgb_led` with `arduino-cli compile --warnings all` (the closest thing to a linter available, since no standalone C/C++ linter is installed on this machine) and writes output to `workspace/lint.log` instead of the terminal. Does not cover `workspace/sketches/` or `workspace/exercises/`.

**Before debugging "the firmware doesn't work," confirm what's actually flashed to the board.** Firmware flashed via `arduino-cli`/the Arduino IDE lives outside git and outside this repo's `docs/` if it was ever compiled from a sketch that isn't checked in here ã the board's own boot-time `Serial` output (visible via `arduino-cli monitor` or any script that reads the port right after reset) names the running sketch and is the source of truth, not an assumption based on what's in this repo. A whole debugging session was once burned chasing a phantom bug in a Python sender script when the real cause was an orphaned, never-committed draft `.ino` left flashed on the board from earlier exploration ã its `setup()` blocked on a full e-paper refresh before entering `loop()`, overflowing the default 256-byte Serial RX buffer and silently truncating every incoming frame, regardless of payload content. `workspace/sketches/test_card/test_card.ino` (see the Architecture section below) has that bug fixed (`Serial.setRxBufferSize()` called before `Serial.begin()`) ã but the lesson applies to *any* sketch, including this one: always confirm what's actually flashed rather than assuming.

## Architecture (current code)

- **`app/`**: implements milestone 1's Android side (USB-serial connect + send) plus the Diagnostics screen, hardware-verified. Single Gradle module, everything flat in the `com.example.crowpanelhub` root package (no `data`/`viewmodel`/`navigation` subpackages) ã a deliberate simplicity choice at this project size, same reasoning as skipping the `Transport` interface (see Transport & wire protocol above). Hilt for DI; `androidx-navigation-compose` provides a two-route `NavHost` (`"main"`/`"diagnostics"`) hosted directly in `MainActivity.kt` rather than a separate nav-graph file, since there are only two screens so far.
  - `CrowPanelHubApp.kt`: `@HiltAndroidApp` `Application` class, registered in the manifest as `android:name=".CrowPanelHubApp"`.
  - `MainActivity.kt`: `@AndroidEntryPoint` `ComponentActivity` hosting a `NavHost` (routes `"main"`/`"diagnostics"`, via `rememberNavController()`) inside a `Scaffold`, rendering `MainRoot()`/`DiagnosticsRoot()`. The template's `Greeting`/`GreetingPreview` composables are gone, fully replaced.
  - `MainViewModel.kt`: `@HiltViewModel`, MVI-style (`MainState`/`MainAction`/`MainViewModel` ã no `Event` type, since navigating to Diagnostics is handled by a plain callback (`onNavigateToDiagnostics`) passed into `MainScreen`, not a one-time emitted event). Observes `UsbSerialTransport.connectionState` and maps it to `MainState.statusText`/`isConnected`/`isBusy`. `onAction` handles `OnTransportSelected`, `OnConnectClick` (calls `UsbSerialTransport.connect()`), `OnSendTextClick` (sends `TextBitmap.generate()`), and `OnSendCheckerboardClick` (sends `Checkerboard.generate()`) ã the two send actions share a private `send(payload)` helper, which also drives `MainState.progressFraction`: since the wire protocol is fire-and-forget with no completion signal from the ESP32, it ticks a simulated progress bar across an estimated write-time-plus-panel-refresh window (~1.3s write + ~4s cooldown) rather than reporting real progress.
  - `MainScreen.kt`: `MainRoot`/`MainScreen` composables (Root + Screen in the same file, per convention) ã a "Diagnostics" button (calls `onNavigateToDiagnostics`), a `SingleChoiceSegmentedButtonRow` transport picker (currently just "USB"), a Connect button, "Send Text"/"Send Checkerboard" buttons (enabled only once connected and not mid-operation), and a status row (`CircularProgressIndicator` while `isBusy`, plus status text). Includes a `@Preview`.
  - `UsbSerialTransport.kt`: `@Singleton`, Hilt-injected with `@ApplicationContext`. Talks to the board via `usb-serial-for-android`'s `UsbSerialProber`. `connect()` finds the device, requests USB permission if not already granted (bridging Android's async permission dialog into a suspend function via `BroadcastReceiver` + `PendingIntent` ã `FLAG_MUTABLE` and `ContextCompat.registerReceiver(..., RECEIVER_NOT_EXPORTED)` both matter here and are easy to get subtly wrong, see inline comments), opens the port at 115200/8N1, and exposes `connectionState: StateFlow<UsbConnectionState>` (`Disconnected`/`RequestingPermission`/`PermissionDenied`/`Connecting`/`Connected`/`ConnectFailed`). `sendImage(payload)` builds the wire frame and writes it. No `disconnect()`/reconnect handling exists ã a documented gap, not an oversight, since there's no Disconnect button in this design. Also owns `logEntries: StateFlow<List<LogEntry>>` (capped at 100, each a timestamp + message), appending an entry on every state transition and send attempt/result ã this feeds the Diagnostics screen's event log. `UsbConnectionState.toStatusText()` used to be private to `MainViewModel`; it's now a top-level function here so both `MainViewModel` and `DiagnosticsViewModel` can share it.
  - `DiagnosticsViewModel.kt`: `@HiltViewModel`, read-only ã no `Action`/`Event` type, since this screen only displays state (`DiagnosticsState`: `statusText`, `lastErrorText`, `logEntries`) and has nothing to trigger. Observes `UsbSerialTransport.connectionState` (mapped via the shared `toStatusText()`, with `lastErrorText` set for `ConnectFailed`/`PermissionDenied`) and `UsbSerialTransport.logEntries` directly.
  - `DiagnosticsScreen.kt`: `DiagnosticsRoot`/`DiagnosticsScreen` composables ã status text, last error (if any), and a `LazyColumn` event log (newest first, each row timestamped via `SimpleDateFormat("HH:mm:ss")`). Includes a `@Preview`.
  - `WireFrame.kt`: pure Kotlin, builds the wire-protocol frame (see Transport & wire protocol above) ã owns the `DISPLAY_WIDTH`/`DISPLAY_HEIGHT`/`PAYLOAD_SIZE` constants (shared by both bitmap generators below) and the local-time-epoch trick.
  - `TextBitmap.kt`: draws demo text onto a 400û300 `Canvas` (via `Paint`) and packs the result into the same 1bpp format `WireFrame` expects ã the phone-side equivalent of `display_text.ino`'s hardcoded rows, but sent as a plain bitmap over the wire protocol rather than rendered firmware-side. Wired to the "Send Text" button.
  - `Checkerboard.kt`: the original milestone-1 test pattern (matches `workspace/sketches/test_card/send_checkerboard.py`'s algorithm), wired to the "Send Checkerboard" button. Kept alongside `TextBitmap` deliberately as a second known-good pattern for future experimentation, not dead code.
  - `ui/theme/{Color,Theme,Type}.kt`: default Material 3 theme (`CrowPanelHubTheme`), unmodified template colors/typography, dynamic color enabled on API 31+.
- **`workspace/sketches/display_text/`** (under `workspace/sketches/`, itself under `workspace/`, a repo-root sibling of `app/`): a standalone Arduino sketch, not wired into the Gradle build. Placed here deliberately as a **reference example** (GxEPD2 init/pin-mapping/power-cycle patterns) ã do not modify or delete it, and do not treat it as a spike to evolve into the milestone-1 firmware; write new sketch files for that instead, consulting this one for patterns.
  - `display_text.ino`: defines a `DisplaySettings`/`Row` model (up to 8 text rows, per-row font size, optional border/color-inversion), a `GxEPD2_BW<GxEPD2_420_GYE042A87, ...>` display instance on pins `PWR=7 BUSY=48 RES=47 DC=46 CS=45`, and an `updateDisplay()` that powers the panel, draws the configured rows, calls `epd.display()`, then hibernates and powers down. `setup()` hardcodes 5 rows of demo text ("Hello World" / "Second line" / ...) and calls `updateDisplay()` once; `loop()` just delays ã no serial input handling, no dynamic content.
  - `config.h`: compile-time constants (display size 400û300, margin, baud rate, font sizes, row heights).
  - `install.sh`: compiles and uploads this sketch, with its own hardcoded FQBN (see `docs/fqbn.txt` bullet above).
  - `README.md`: a code walkthrough plus the "Build and flash" commands ã no flowchart/generator script here, unlike the other firmware folders, since this one stays intentionally minimal as a reference (see above).
- **`workspace/sketches/receive_image/` no longer exists.** It was the original milestone-1 firmware sketch ã a serial receive loop, not present in `workspace/sketches/display_text/` ã and had been flashed to and verified against real hardware (a CrowPanel board on `/dev/ttyUSB0`, CH340 bridge chip) via both hand-written Python scripts and the real Android app before it was removed. It was deleted because it wasn't a clear teaching vehicle (see this file's "Teaching mode" section under Working style, above); `workspace/sketches/test_card/` (below) implements the same wire protocol today. If you see it mentioned in old context, treat it as superseded by `workspace/sketches/test_card/`.
- **`test_screen/` no longer exists.** An earlier, in-progress sketch by that name was never actually committed to this repo beyond two files (`generate_test_pattern.py`, `CompletePattern.jpg`) ã everything else this document once described there (`test_screen.ino`, its `config.h`, its README/flowchart, `render_pm5540_preview.py`) was aspirational and never made it to disk. Those two real files have since been moved into `workspace/sketches/test_card/` below, and `test_screen/` was deleted. If you see it mentioned in old context, treat it as superseded by `workspace/sketches/test_card/`.
- **`workspace/sketches/test_card/`** (sibling of `workspace/sketches/display_text/` within `workspace/sketches/`): a second, independent firmware sketch, and a board only ever runs whichever one was flashed last (see the "confirm what's actually flashed" note above). Compiles for `esp32:esp32:esp32s3` and **has been flashed to and verified against real hardware** (a CrowPanel board on `/dev/ttyUSB0`, CH340 bridge chip): after `arduino-cli upload`, the boot self-test was confirmed printing `test_card ready, waiting for frames`, and `generate_test_pattern.py --send` confirmed a real wire-protocol frame is received, validated, and rendered ã the board printed `frame ok, Last updated: ...` and the panel updated accordingly. Draws its own built-in Philips PM5544-style test card on boot (no phone/PC needed) *and* accepts wire-protocol frames over Serial afterwards. Has the `Serial.setRxBufferSize()` fix (called before `Serial.begin()`, sized to survive the boot self-test render) referenced in the "confirm what's actually flashed" note above.
  - `test_card.ino`: per-pixel band/border color functions ported directly from `generate_test_pattern.py`'s `draw_top_split()`/`draw_grating()`/etc. (see the file's comments), plus a from-scratch Floyd-Steinberg disperser (`renderPM5544Card()`) that dithers straight into the packed output buffer one row at a time using two small 400-entry error rows, rather than ever allocating a full 120,000-byte grayscale canvas the way Pillow's `Image.convert("1")` does. Its boot self-test reserves a bottom label strip (circle centered at `CONTENT_HEIGHT/2`, i.e. `DISPLAY_HEIGHT` minus the strip) ã a deliberately different layout from `generate_test_pattern.py`'s full-canvas card below, since that one is meant for wire-protocol-frame rendering instead.
  - `install.sh`: compiles and uploads this sketch, with its own hardcoded FQBN (see `docs/fqbn.txt` bullet above).
  - `config.h`: identical wire-protocol constants and pins to the other sketches' `config.h`. The single source of truth for those constants on the Python side too ã see `config_h.py`.
  - `config_h.py`: a small regex-based parser that reads `config.h`'s `const <type> NAME = <expr>;` lines (evaluating simple expressions like `PAYLOAD_SIZE`'s formula) and exposes them as Python values, so `generate_test_pattern.py`/`render_preview.py` never hardcode a value `config.h` already defines. Not a general C header parser ã just enough for this one file's actual shape.
  - `serial_sender.py`: the low-level module standing in for the Android sender during firmware bring-up (before the Android app existed) ã stdlib-only (no `pyserial`), resets the board via raw `termios`/`fcntl` DTR/RTS control, builds a frame via `build_frame(payload)` with the current **true UTC** time (unlike the Android app, this never adds a local offset), and exposes `send_payload(payload)` (sends it, prints the board's `Serial` responses). Not run directly.
  - `send_checkerboard.py`: the simplest sending script ã builds a fixed checkerboard payload and calls `serial_sender.send_payload(...)`. This folder's own `generate_test_pattern.py` imports `send_payload` from the local `serial_sender.py` too, so `test_card/` compiles, flashes, and runs its Python tooling standalone with the folder copied in isolation (verified).
  - `generate_test_pattern.py`: builds the PM5544-style test card as a full-canvas payload (circle centered on all 300 rows, no reserved label strip) and sends it over the wire protocol via `--send`, reusing this folder's own `serial_sender.py`'s `send_payload()` (pulling `DISPLAY_WIDTH`/`DISPLAY_HEIGHT`/`PAYLOAD_SIZE` from `config_h.py` instead). Moved here from the now-deleted `test_screen/` (see above).
  - `CompletePattern.jpg`: reference photo of a real PM5544 card, moved here from `test_screen/`, used to measure `generate_test_pattern.py`'s band proportions.
  - `render_preview.py`: a line-for-line Python port of `test_card.ino`'s own boot self-test drawing code (distinct layout from `generate_test_pattern.py`, see above) ã for checking the card/label placement without hardware. Saves to `boot_preview.png`, deliberately not `preview.png`, so the two scripts' outputs can't collide. Pulls `DISPLAY_WIDTH`/`DISPLAY_HEIGHT` from `config_h.py`; its other layout constants (`BORDER_CELL`, `CIRCLE_RADIUS`, `LABEL_HEIGHT`, etc.) are hand-kept in sync with `test_card.ino`'s own copies instead, since those live in the `.ino` itself, not `config.h` (see that file's comment on why).
  - `README.md` / `generate_flowchart.py` / `flowchart.png`: same documentation approach as the other sketches'.

## Build configuration notes

- Kotlin 2.2.10, AGP 9.2.1, Gradle 9.4.1 (see `gradle/wrapper/gradle-wrapper.properties`).
- compileSdk/targetSdk 37, minSdk 24. `sourceCompatibility`/`targetCompatibility` are JavaVersion 11 (not 17).
- Compose compiler is applied via the `org.jetbrains.kotlin.plugin.compose` plugin (Kotlin 2.x style), not the old `composeOptions.kotlinCompilerExtensionVersion`.
- **Hilt** (`com.google.dagger.hilt.android`, version `2.59.2`) for DI, with **KSP** (`com.google.devtools.ksp`, version `2.3.9`) for annotation processing ã not kapt; use the `hilt-compiler` artifact with `ksp(...)`, not the kapt-era `hilt-android-compiler`. KSP's version here is **not** tied 1:1 to the Kotlin version ã KSP switched to a standalone versioning scheme at `2.3.0`. `2.3.9` specifically was required because older KSP builds paired to Kotlin 2.2.10 (e.g. `2.2.10-2.0.2`) don't correctly support AGP 9's "built-in Kotlin" compilation (`google/ksp#2729`); if bumping Kotlin or KSP later, re-verify built-in-Kotlin compatibility rather than assuming an old pairing still works ã a mismatch here fails in a confusing way (`compileDebugKotlin`/`kspDebugKotlin` tasks silently don't run, so the build reports success while actually producing an APK missing all Kotlin-compiled classes).
- **usb-serial-for-android** (`com.github.mik3y:usb-serial-for-android`, version `3.10.0`) is fetched from **JitPack** ã `maven(url = "https://jitpack.io")` is added to `settings.gradle.kts`'s `dependencyResolutionManagement.repositories`, not inside `app/build.gradle.kts` (a per-module `repositories {}` block there is disallowed by this project's `FAIL_ON_PROJECT_REPOS` mode).
- **androidx-navigation-compose** (version `2.9.8`) provides the `NavHost` wiring `MainScreen`/`DiagnosticsScreen` together ã added alongside the pre-existing `androidx-hilt-navigation-compose` (which was already present for `hiltViewModel()` regardless of navigation).
- No `kotlinx.serialization` plugin is applied anywhere in the project yet.
- Dependency versions are centralized in `gradle/libs.versions.toml` (version catalog) ã add new dependencies there rather than hardcoding coordinates in `app/build.gradle.kts`.
